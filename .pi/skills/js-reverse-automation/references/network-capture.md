# 网络捕获规则

本规则用于在不改变原有主链路的前提下，强化 Phase 1 的请求侦察质量。目标是先稳定锁定目标请求，再进入源码定位和 Hook。

## 适用时机
- 需要确认目标参数到底出现在 URL、Header、Cookie、JSON body 还是表单 body。
- 页面动作会触发多个相似请求，必须先缩小到真正参与签名或加密的那个请求。
- 用户给了 `Optional Fetch Example`，但需要映射到真实浏览器中的对应请求。

## 证据优先级
1. 真实页面动作触发后的 Network 记录
2. 请求详情中的 method、URL、headers、payload、response
3. 页面内观测代码打印出的调用栈、序列化对象或函数参数
4. 控制台日志与页面状态快照

## 推荐流程
1. 打开目标页面，完成必要交互，确保请求真实发出。
2. 用 chrome-devtools-mcp 的网络请求列表先筛出候选请求。
3. 对候选请求逐个读取详情，确认目标参数落点与请求体形态。
4. 如果同类请求很多，优先按以下特征缩小范围：
   - 目标参数是否出现
   - 请求触发时间是否贴近用户动作
   - 响应状态是否符合业务预期
   - 请求体是否包含未加密明文和加密结果的邻近字段
5. 一旦锁定请求，立刻记录：
   - 请求 URL
   - HTTP 方法
   - Content-Type
   - 参数位置
   - 关键 Header
   - 触发动作
6. 如果仅靠请求详情不足以定位源码，再进入最小观测代码或最小 Hook。

## 成功标准
- 至少有一个请求被明确标记为目标请求。
- 已知每个待分析参数的落点。
- 已记录能够支持后续源码定位的请求上下文。

## 失败信号
- 只能看到静态资源请求，看不到业务 API。
- 目标动作会触发多个高度相似请求，但无法分辨哪一个参与签名。
- 有请求，但无法确认目标参数出现在哪一层序列化之后。

## 处理策略
- 对多请求竞争场景，增加一次单变量实验：
  - 只改一个输入值，再看哪个请求字段同步变化。
- 对序列化不透明场景，补最小 Hook 到：
  - `window.fetch`
  - `XMLHttpRequest.prototype.open`
  - `XMLHttpRequest.prototype.send`
  - `JSON.stringify`
- 这些观测点通过 `evaluate_script` 注入；如果页面会刷新，可通过 `navigate_page(initScript=...)` 在下一次导航前预注入。
- Hook 的目的只是把“发包前最后一次可观测形态”记录下来，不是替代网络证据。
