# 协议层约束验证规则

本规则用于判断“算法已经基本定位，但请求复现或脱离页面上下文调用仍失败”时，问题是否来自协议层约束而不是算法本身。

## 适用时机
- 浏览器内请求成功，但离开页面上下文后失败。
- 相同参数在页面中正常，在本地代理或手工重放时失败。
- 请求是否发出并不是问题，但服务端返回值表现出环境、会话或时序依赖。

## 优先检查项
1. 请求方法、路径、查询参数和 `Content-Type`
2. 必需 Header
3. Cookie、localStorage、sessionStorage 或首包预热状态
4. `Origin`、`Referer`、CSRF 等上下文约束
5. 请求顺序、时间窗口、nonce、timestamp 和频率限制
6. 浏览器中是否暴露出 HTTP/2 或其他连接层差异；如果当前工具拿不到足够证据，只能记录为待验证风险

## 工作原则
- 先证明“浏览器里为什么成功”，再判断“脱离页面后为什么失败”。
- 优先记录差异，不要直接把所有浏览器请求头照搬进最终产物。
- 只有真正参与校验的协议层条件，才应进入 `analysis_result.json` 的风险或依赖描述。
- 对 `chrome-devtools-mcp` 无法直接观测的连接层细节，只能做差异推断，不能写成已确认事实。

## 推荐流程
1. 用真实浏览器请求作为基线，记录 URL、方法、Header、Cookie、响应状态和响应特征。
2. 观察同一路径是否依赖前置请求，例如：
   - 首页或接口预热
   - 动态 Cookie 下发
   - CSRF token 初始化
   - 首次请求建立会话
3. 对失败请求做最小差异对比，确认失败是由以下哪类因素触发：
   - 缺失 Header 或 Cookie
   - 顺序错误
   - 时间窗过期
   - 频率限制
   - 页面上下文外的协议差异
4. 只有当这些因素会影响最终 JSRPC / Flask 生成与验证时，才记录到最终产物。

## 可接受结论
- “算法正确，但仍依赖页面中的会话或预热状态”
- “算法与请求体正确，但还依赖某个 Header / Cookie / token”
- “脱离页面上下文失败，疑似存在 TLS/HTTP2/连接层特征差异”

## 禁止行为
- 未经验证就把协议层失败归因为算法失败。
- 为了提高成功率，把整份浏览器请求头或无关状态原样固化进生成代码。
- 在无法验证 TLS/HTTP2 差异时，伪造确定性结论；应明确写成风险或残余限制。
