# 源码定位规则

本规则用于把 Phase 1 锁定的目标请求，反推到 Phase 2 需要的源码位置和候选入口。

## 核心原则
- 先从请求详情、页面内调用栈和序列化节点回溯，再做关键词搜索。
- 先找“参数最后一次变化的位置”，再找“算法最深处”。
- 只接受带证据的源码定位结果，不接受纯文本匹配结论。

## 定位优先级
1. 请求详情对应的脚本线索、页面内 `Error().stack` 或观测代码捕获的栈帧
2. 发包前对象构造或序列化点
3. 与参数同名或邻近字段的源码片段
4. 加密库调用点

## 推荐流程
1. 从目标请求出发，先读取请求详情，确认请求方法、请求体和相关脚本线索。
2. 对请求构造点补最小观测代码：
   - 观测 `fetch` / XHR `send`
   - 观测 `JSON.stringify`
   - 在页面内输出 `Error().stack`
3. 如果页面已刷新或将跳转：
   - 用 `navigate_page(initScript=...)` 预注入观测代码
   - 再次复现目标动作并记录序列化前对象
4. 识别参数的三个关键位置：
   - 明文进入点
   - 加工或加密点
   - 发包前最终写入点
5. 只有当这三个位置中的至少两个被证据串起来，才可输出首选入口。

## 候选入口接受条件
- 能解释目标参数从明文到密文的变化。
- 能定位到可调用函数或稳定 resolver。
- 能说明 `this` 绑定、参数签名和必要依赖。

## 常见误判
- 只命中参数名，但该变量只是中间拷贝。
- 只命中 `md5` / `aes` / `sha`，但它并不直接服务目标请求。
- 只看到包装函数，没有继续确认真实执行点。

## 输出要求
- `source_hint` 要尽量落到具体 bundle 位置或对象路径。
- `evidence` 至少包含以下之一：
  - 页面内调用栈
  - Hook 捕获的入参与返回值
  - 序列化前对象快照
  - 参数值对比实验结果
